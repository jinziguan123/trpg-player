# 战斗轮重构设计（2026-07-11 · v2）

> 状态：**设计已定稿（本次会话与用户逐条敲定）**，待落实施计划后写代码。
> 前身：[2026-07-10-战斗轮与追逐设计.md](2026-07-10-战斗轮与追逐设计.md)（P1–P5 已落地：先攻队列/引擎结算/子代理叙述/前端面板/结果折回）。
> 本文只描述 **v2 的增量与改动**，未提及处沿用 v1。

## 关键决策（已定）

1. **防御 = 交互式反应**：被攻击的**真人**当场抉择 `反击 / 闪避 / 扑掩体`；NPC 与 AI 队友的防御由子代理/启发式自动决定，不弹窗。战斗因此从「一个请求跑到底」改为 **可暂停/可续跑的状态机**。
2. **删除 `_default_defense` 启发式**——它是「闪避成功攻击者却扣血」（下称 ⑤）的病灶。
3. **火器不可反击**：被火器攻击的 `allowed` 只含 `闪避/扑掩体`（RAW 一致）。火器 + 闪避口径 = **射手转困难难度射击**（扑向掩体的无坐标简化）。
4. **可视化战场 = HUD 式、非空间**：我方/敌方两栏卡片 + 顶部先攻轨 + 折叠战斗日志抽屉 + KP 叙述凸显。**不引入坐标/瓦片/站位**（承 2026-07-04 删除场景小地图的教训）。HP 变化配**动画提示**（受伤红闪 `-N`、治疗绿涨 `+N`）。
5. **动作集 v1（完整版）**：基线 `攻击 / 逃跑` + `急救·医学 / 观察战场 / 擒抱·缴械 / 装填·瞄准`；反应 `反击 / 闪避 / 扑掩体`。
6. **状态子系统 = 声明式、确定性引擎**：把散落的状态逻辑抽成数据驱动的 `StatusEffect` 层（插件式），带生命周期钩子。**不套 LLM planner**（理由见 §2）。
7. **暂停/续跑落地方式**：`pending_reaction` 落进 `world_state.combat`，靠新端点续跑；**不**挂起 HTTP/长轮询（抗断线重连）。
8. **Bug ①（校验器 JSON 解析失败）** 与战斗重构**解耦**，作为 P0 独立快修。

## 2. 为何不把战斗引擎做成 turn planner 架构

用户提出「战斗引擎是否该仿 turn planner」。结论：**机制部分不该，这是错配**。

- **turn planner 驯服模糊**：自由输入 + 叙事上下文 → 要不要检定/揭什么线索/NPC 怎么反应，是灰色判断，故让 LLM 提议、结构化契约约束。
- **战斗执行确定**：伤害、对抗、半血阈值、昏迷检定、状态迁移全是规则书写死的纯函数。用 LLM「规划」它 = 更慢、更不可靠（算术/阈值最不该交给 LLM）、更难测，且违背 CLAUDE.md「规则逻辑走**确定性** RuleEngine」。
- **拆开 turn planner 的两个价值**：(a) 结构化数据契约、(b) 由 LLM 生成它。战斗**要 (a) 不要 (b)**——`ActionOutcome`/`RoundResolution` 契约（记录谁打谁/检定/伤害/触发状态）对日志·evals·前端都有用，但由**确定性结算产出**。
- **战斗里「该用 LLM 的那半」已是 turn-planner 同构**：`CombatAgent.decide` 产结构化战术动作 → `_validate_npc_action` 容错校验 → 非法回落启发式（[combat_service.py:326](../../server/app/services/combat_service.py)）。粒度精准锁在「NPC 该做什么」这个唯一模糊点，不再外扩。

## 3. 核心模型：可暂停的战斗状态机（修 ④⑤）

`world_state.combat` 新增字段：

```jsonc
{
  // …v1 既有字段（active/round/turn_index/initiative/log/flee_to_chase）…
  "pending_reaction": {                 // 非空 = 战斗驱动已暂停，等某真人防御者选反应
    "attacker_id": "enemy_ab12cd",
    "defender_id": "char_xxx",
    "weapon": "徒手格斗",
    "ranged": false,
    "allowed": ["fight_back", "dodge", "cover"]   // ranged=true → 去掉 fight_back
  }
}
```

驱动流转改动（[drive_npcs](../../server/app/services/combat_service.py)）：

- 当某行动者攻击一个**真人且存活**的防御者 → **不再** `_default_defense` 自动结算，而是：置 `pending_reaction` → 广播 `combat_reaction_prompt` chunk → `_save_combat` → **return（停止驱动）**。
- 新端点 **`POST /sessions/:id/combat/reaction`** `{choice: fight_back|dodge|cover, weapon?}`：
  - 校验提交者 == `pending_reaction.defender_id` 且 choice ∈ allowed；
  - 用所选防御结算这一击（`resolve_attack(defense=choice)`）→ 应用伤害/状态 → 清 `pending_reaction`；
  - 从暂停点**续跑** `drive_npcs`，直到下一个真人*行动*回合或战斗结束。
- **NPC/AI 队友当防御者**：不弹窗，防御由攻击结算时就地取子代理/启发式的选择（沿用现路径，只是防御选择改为显式传入而非 `_default_defense`）。
- **玩家攻击 NPC**：NPC 作为防御方自动选反击/闪避（AI），单次对抗解决，无嵌套弹窗（反击的还手是对抗结算的一部分）。
- **死锁保护（可选，默认关）**：反应提示可带「本回合默认闪避」快捷开关，多敌连攻时免反复弹窗，也防真人 AFK 卡场。

火器（[combat.py resolve_attack](../../server/app/rules/coc/combat.py)）：
- `ranged=true` 且 `defense=dodge/cover` → 射手改判**困难难度**（扑向掩体）；命中即伤，无反击。
- 移除「远程 = 完全无防御」的旧行为。

## 4. 声明式状态子系统（补：角色状态处理）

把状态逻辑从 `apply_damage` 内联中抽出，做成**数据驱动的 StatusEffect 层**（纯 `combat.py`，插件式）：

```python
# 形态示意（纯函数、可 monkeypatch 钉死骰子）
@dataclass
class StatusEffect:
    key: str                    # major_wound / dying / unconscious / grappled / disarmed / bleeding / burning
    # 生命周期钩子（引擎在对应时点调用，返回结算行 + 状态变更）
    on_apply:       Callable | None   # 获得时：如 major_wound → CON 检定失败则转 unconscious
    on_round_start: Callable | None   # 每轮开始 tick：如 dying → 每轮 CON/急救稳住否则恶化；bleeding → 掉血
    on_defense:     Callable | None   # 防御修正：如 grappled → 不能闪避
    on_attack:      Callable | None   # 攻击修正：如 disarmed → 不能用武器
```

覆盖矩阵：

| 状态 | 触发 | 钩子行为 |
|---|---|---|
| `major_wound` 重伤 | 单击 ≥ 半血 | `on_apply`: **CON（体质）检定**失败 → 转 `unconscious`（RAW，非意志/POW） |
| `dying` 濒死 | HP ≤ 0 | `on_round_start`: 每轮 CON 掷或急救稳住，否则恶化/死亡 |
| `unconscious` 昏迷 / `dead` 死亡 | 派生 | 出局判定（`is_active` 已处理） |
| `grappled` 擒抱 / `disarmed` 缴械 | 机动动作胜 | `on_defense`: 不能闪避 / `on_attack`: 无法用武器 |
| `bleeding` 流血 / `burning` 燃烧 | 武器 flag（现已收在 `roll_weapon_damage.flags`） | `on_round_start`: 每轮伤害。**可选，YAGNI 可缓** |
| SAN / 临时疯狂 | 理智检定失败 | 另一条线。**可选，YAGNI 可缓** |

分工：**规则判断**（是否重伤/掷哪个检定/落什么状态）进纯 `combat.py`；**落库/写回角色卡/广播**留 `combat_service`。拆开现 `apply_damage` 的混合。状态机在两个时点驱动状态：`apply_damage` 后（on_apply）、每个行动者回合开始（on_round_start）。

## 5. 动作集扩展（③）——纯函数，好测

行动类型扩为 `attack | first_aid | observe | maneuver | reload | aim | flee | wait`；反应 `fight_back | dodge | cover`。

| 动作 | 判定 | 结算 |
|---|---|---|
| `first_aid` 急救/医学 | 急救 or 医学检定 | 成功回 1 HP / 稳住濒死；每处伤一次（RAW） |
| `observe` 观察战场 | 侦查 or 心理学检定 | 成功 → 产一条情报 beat 交子代理描述敌情/弱点（不改数值） |
| `maneuver` 擒抱/缴械 | 对抗格斗（含体格差） | 不造伤害，施加 `grappled`/`disarmed` 状态 |
| `reload` 装填 / `aim` 瞄准 | 无 | 置 `loaded`/`aim` 标记；瞄准给下一击 1 个奖励骰 |

每个动作在引擎产出结构化 `ActionOutcome`（谁/对谁/检定/伤害/状态变更），供日志/evals/前端统一消费。

## 6. 前端 HUD 战场（② ⑥）

新组件 **CombatStage**（增强/替换 [CombatPanel.tsx](../../apps/web/src/components/game/CombatPanel.tsx)）：

- **我方 / 敌方两栏卡片**：图标（`react-icons/gi`，禁 emoji）、HP 条 + 数字、状态徽标、武器。
- **顶部先攻轨**：横排行动顺序，高亮「当前」、标「下一个」，走过者淡化 → 解决①「顺序不直观」。
- **HP 变化动画**：受伤 → `-N` 红字上浮 + 血条红闪 + 卡片轻抖；治疗 → `+N` 绿字上浮 + 血条绿涨。复用现有「骰子落定才显结果卡」的动效节奏与 reduced-motion 降级。
- **反应提示**：轮到自己被攻击时卡片处弹出 `反击[火器禁用] / 闪避 / 扑掩体` + 一句情境（`X 用 Y 攻击你`）。
- **主动动作栏**：攻击（目标/武器）/ 急救（选队友）/ 观察 / 擒抱缴械（目标）/ 装填瞄准 / 逃跑。
- **折叠战斗日志抽屉**：检定行/伤害行/状态变化收进可展开面板（默认折叠、只露最近 1~2 条摘要），**不再灌满主聊天流** → 解决②。做法：后端给这些 chunk 的 metadata 打 `combat_log:true`（仍落库供复盘/重连），前端据此归入抽屉。
- **KP 叙述**：仍作为 `narration_full` 在主叙事区凸显（保持 v1）。

## 7. Bug ① 校验器快修（P0，独立）

[turn_validator.py](../../server/app/ai/turn_validator.py) 改契约与解析：
- `violated=false` 时**不要求** `corrected_narration`（多数情况输出极短，不会截断）；只有 `violated=true` 才要改写文本。
- `complete` 复用「内部流式」（见 commit d647ba4）避免长输出被掐断；`max_tokens` 给足（叙述长度 ×2 + 缓冲）。
- 更稳的 JSON 提取：剥 ```json 围栏、截取首个平衡 `{...}` 再 `json.loads`。

## 11. 风险与未决

- **交互反应的联机等待**：真人防御时其他玩家等待；v1 接受，配「本回合默认闪避」缓解 AFK 卡场。
- **火器+闪避口径**：先用「射手转困难」简化；若手感不佳可改「闪避成功即完全避开」。
- **状态子系统边界**：v1 落重伤/濒死/昏迷/擒抱/缴械；流血/燃烧/SAN 标为可选，按需排期，默认 YAGNI 缓。
- **续跑与广播时序**：反应端点结算后续跑可能产出多条 chunk，需保证顺序与前端日志抽屉的归并一致。
