# 场景配图视觉状态设计

> 2026-07-20。目标：让场景图片表达“当前场景的视觉状态”，同时保持跑团节奏、缓存命中率和多人会话隔离。

## 1. 玩家体验契约

- 首次进入场景时展示基础场景图；图片生成异步进行，不阻塞叙事。
- 同一会话再次进入同一场景、且视觉状态没有变化时，不重复创建配图卡。
- 场景发生明确的视觉变化时，创建一张新的状态配图卡；旧卡保留，保证历史回放一致。
- 普通线索发现、检定成败、内部剧情标志变化，如果没有视觉影响，不生成新图。
- 图片文件失效时修复原卡，不创建第二张重复卡；生成失败时剧情继续，并保留可重试入口。

## 2. 状态模型

场景视觉身份由三部分组成：

```text
session_id + scene_id + visual_state_key
```

`visual_state_key` 的来源按优先级排列：

1. 场景解析结果中的显式 `visual_variant`，用于作者明确命名状态，如 `flooded`、`lights_on`。
2. 场景状态变体覆盖的视觉字段摘要。当前纳入：标题、描述、危险度、氛围和地图。
3. 没有视觉变化时使用 `base`。

基础图片保存在 `scenes[].image`。状态变体保存在同一场景条目的 `image_variants` 对象中：

```json
{
  "id": "scene_6",
  "image": "/api/images/base.jpg",
  "image_variants": {
    "flooded": "/api/images/flooded.jpg"
  }
}
```

动态状态图片不得覆盖基础图片。模组级缓存可以复用作者定义的稳定状态；由玩家临时行为产生的状态仍由会话的 `visual_state_key` 隔离，不能影响其他会话的当前状态。

## 3. 触发与去重

场景图入口是开场、显式前往、确定性场景切换，以及剧情标志导致的当前场景视觉状态变化。入口调用统一的场景配图函数，由它计算视觉状态键并决定：

- 状态卡已存在且图片有效：直接跳过。
- 状态卡不存在且缓存有效：落卡并直接带图。
- 状态卡不存在且缓存无效：先落卡，再后台生成并通过 `event_patch` 补图。
- 状态卡已存在但图片失效：查找原事件并后台修复，禁止重复落卡。

旧会话中的 `scene_cards: ["scene_id"]` 仍按基础状态卡兼容处理；新状态卡使用 `scene_id|visual_state_key` 作为去重标识。

## 4. 图片失效与重试

图片 URL 只有在本地文件存在、文件名符合 `/api/images/<uuid>.jpg` 等约束时才算有效。前端加载失败时调用图片修复接口，并携带 `visual_state_key`；后端只修复对应基础图或对应状态变体。

同一会话、同一状态键同时只允许一个后台生成任务。任务失败不会阻塞剧情，也不会清除状态卡；用户可以重新加载或点击修复入口再次尝试。

## 5. 不触发生图的场景

- 仅新增或升级线索台账。
- 仅进行技能检定、理智检定或伤害结算。
- 仅设置没有视觉字段变化的剧情 flag。
- 仅修改 KP 内部备注、规则说明或幕后状态。

模组编辑时，只有标题、描述、危险度、氛围、地图等视觉字段改变，才应视为基础图缓存过期；其他字段修改不影响图片缓存。

## 6. 验收标准

1. 同一会话同一场景基础状态只出现一张场景卡。
2. 场景状态从基础变为 `flooded` 后出现第二张状态卡，回到 `flooded` 时不再重复。
3. 删除基础图或状态图文件后，原卡被修复，事件数量不增加。
4. 生图失败时仍能继续跑团，重试不会并发生成多张图。
5. 不同会话的动态状态不会覆盖基础图，也不会把一个会话的临时状态泄漏到另一个会话。
