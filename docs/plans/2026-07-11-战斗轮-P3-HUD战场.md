# 战斗轮 P3：HUD 战场视图 实施计划

> 承 [2026-07-11-战斗轮重构-design.md](2026-07-11-战斗轮重构-design.md) §6。P1（状态机+反应）、P2（后端动作集）已上线。**直接在 master 开发**（用户已授权）。
> **范围决策**：P2 的前端按钮并入本阶段——CombatStage 的「主动动作栏」一次性含全部动作，不在旧 CombatPanel 上叠加后再重建。

**Goal:** 把战斗从「薄血条卡片带 + 机械结算灌满主聊天流」升级为游戏式 HUD 战场：两栏参战方卡片 + 先攻轨 + 折叠战斗日志抽屉 + HP 变化动画；KP 叙述仍在主叙事区凸显。

**Architecture:** 后端给战斗机械 chunk（dice/system 结算行）打 `combat_log:true` 标记（仍落库供复盘/重连）；前端新组件 `CombatStage` 取代现 `CombatPanel` 的渲染，机械结算归入可折叠抽屉、不再进主聊天流；HP 变化由 `combat_state` 前后 HP diff 驱动动画。

**Tech Stack:** React + TS + Tailwind + CSS 变量（哥特暗色 token，见 index.css :root）；图标 `react-icons/gi`（禁 emoji）；动效复用现有「骰子落定才显卡」的节奏 + `prefers-reduced-motion` 降级。

**通用纪律:** 前端 commit 前 `cd apps/web && npx tsc --noEmit && npx vite build` 通过；后端改动 `cd server && .venv/bin/pytest -q && .venv/bin/ruff check .` 全绿；commit 中文 + `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`；分逻辑单元多次 commit。

## Part A — 后端：战斗机械 chunk 打 combat_log 标记

`combat_service.py`：`_combat_line`（system 结算行）、`_combat_dice`（对抗骰行）产出的 chunk 的 metadata 加 `combat_log: true`。**KP 叙述**（`_combat_narration` → `narration_full`）**不打**该标记（它要进主叙事区）。落库事件不变（复盘/重连仍完整）。补/改测试断言这些 chunk 带 `combat_log`。

> 口径：前端据 `metadata.combat_log` 决定「进战斗日志抽屉」还是「进主消息流」。战斗结束/非战斗的普通 dice/system 不受影响（无此标记）。

## Part B — 前端：`CombatStage` 组件（取代 CombatPanel 渲染）

新建 `apps/web/src/components/game/CombatStage.tsx`（可保留/内联现 CombatPanel 逻辑再扩展）。`GameSessionPage` 用它替换现 `CombatPanel` 挂载点。结构：

### B1 先攻轨（顶部）
横向排 `combat.order`，按先攻序；高亮 `combat.turn`（当前）、标「下一个」、走过者淡化；每格显示名字 + 阵营色点 + 我方标记。解决①「顺序不直观」。

### B2 两栏参战方卡片（我方 / 敌方）
- 左栏 side∈{player,ally}，右栏 side==enemy。
- 每卡：图标（SeatIcon 风格 / gi）、名字、HP 条+数字、`status` 徽标（重伤/濒死/昏迷/死亡/逃离，复用现 STATUS_META）、**conditions 徽标**（grappled「被擒」`GiGrab`、disarmed「缴械」`GiBrokenaxe` 或近义 gi）、武器名。
- 出局（dead/fled）灰化。

### B3 HP 变化动画（你明确要的）
- `CombatStage` 记住上一帧各参战方 HP；新 `combat_state` 到达时对每个 id 做 diff：
  - 掉血（Δ<0）→ 卡片上浮红字 `-N` + 血条红闪 + 轻微抖动；
  - 回血（Δ>0）→ 上浮绿字 `+N` + 血条绿涨。
- 用 CSS 动画/Framer（若项目已有）或纯 CSS keyframes；`prefers-reduced-motion` 时跳过动画只更新数值。与现有 3D 骰子动效节奏一致。

### B4 反应提示（P1 已有雏形，迁入 CombatStage）
`pendingReaction.defender_id===myCharId` → 底部显示情境 + 按 `allowed` 渲染 反击/闪避/扑掩体 按钮（`GiCrossedSwords/GiShield/GiBrickWall`）→ `POST /combat/reaction {choice}`。非我 → 「等待 X 反应…」。

### B5 主动动作栏（并入 P2 动作，仅我方回合）
按钮（gi 图标，缺则换近义）：攻击（目标敌方+武器）、急救（目标己方受伤者）、观察、擒抱/缴械（目标敌方 + kind=grapple/disarm）、装填、瞄准、逃跑。提交走既有 `api.post('/sessions/:id/combat/action', {type, target_id?, weapon?, kind?})`；目标选择器按动作切己方/敌方候选；被擒抱（自己 conditions 含 grappled）时隐藏逃跑/闪避类无效项（可选打磨）。

### B6 折叠战斗日志抽屉
- `GameSessionPage` chunk 路由：`metadata.combat_log===true` 的 dice/system chunk **不进主消息流**，改推入一个战斗日志数组（传给 CombatStage）。
- 抽屉默认折叠，标题「战斗日志 (N)」+ 展开箭头（`GiScrollUnfurled`）；折叠时只露最近 1~2 条摘要；展开显示全部（本场）。解决②「灌满主聊天流、看不懂」。
- **KP 叙述**（narration_full）照常进主叙事区凸显——不动。
- 战斗结束清空日志抽屉（combat_end）。

## 检验
- 后端 pytest：`_combat_line`/`_combat_dice` chunk 带 `combat_log`，`_combat_narration` 不带。
- 前端 `tsc --noEmit` + `vite build` 通过。
- **preview 实机验证**（由我做）：构造/触发一场含 combat_state + combat_log + combat_reaction_prompt 的战斗，确认：先攻轨顺序正确、掉血红字动画、机械结算进抽屉而非主流、反应按钮可用、KP 叙述仍在主区。截图为证。

## 风险 / 未决
- HP diff 动画依赖 combat_state 稳定携带各 id 的 hp（现 `_combat_meta.order` 已带）——确认续跑/重连时 diff 基准不错乱（重连首帧不应把满血误判为回血：首次收到 order 时只建基准、不动画）。
- 日志抽屉与「分头行动」等既有渲染分支的兼容（战斗时假设全队同场，见 P1 风险）。
- 守方持械反击（引擎徒手硬编码）仍留待引擎单独接，不在本阶段。
