# 首团闭环、可维护性与分发文档设计

## 背景与目标

新用户要完成第一局游戏，目前需自行理解 AI 配置、模组、角色与房间的关系。目标是在不提供离线假剧情、也不依赖未经确认授权内容的前提下，让用户从首页进入一个可开始的自制示例单人会话。

本分支同时处理首团路径触及的前端结构债，并补齐项目的许可证、内容声明和产品边界文档。

## 范围

### 首团闭环

首页新增“体验新手团”入口。

1. 用户点击入口后，客户端先检查是否存在可用 AI 配置。
2. 没有配置时，跳转到设置页并保存 `onboarding` 返回意图。
3. 用户完成并测试配置后，设置页将用户带回首团流程。
4. 已配置时，后端在同一原子流程中取得自制示例模组、创建预设调查员、创建单人会话，并返回会话标识。
5. 前端跳转到会话页，让既有开场逻辑生成第一段叙事。

示例内容必须明确标为项目自制内容。该流程不会静默使用现有种子库中的模组或规则书，也不实现离线模拟叙事。

### 前端结构收口

仅拆分首团路径直接涉及的边界：

- `GamePage`：房间列表、新建会话、加入房间、模组筛选与创建会话逻辑。
- `CharacterPage`：把可复用的预设角色创建能力放到独立的 API/服务边界；完整七步车卡的业务行为保持不变。
- 为首团关键路径建立前端测试基础，并为后端编排 API 增加覆盖。

`chat_service.py`、战斗、追逐、地图和完整角色编辑流程不在本分支重构范围内。

### 文档与分发边界

- 增加代码许可证。
- 增加第三方内容与用户导入内容声明。
- 重写 README，按“已实现 / 实验性 / 规划中”说明能力，并明确桌面本地或可信局域网的部署边界。
- 保留现有种子内容，但将“移除或替换未获授权内容”设为公开分发前的阻断项。

## 架构与数据流

首团编排由后端服务负责，避免前端连续调用“找模组、建角色、建会话”时留下半成品。服务返回已创建的会话及首团所需的摘要；失败时整个数据库事务回滚，前端仅显示可行动的错误。

前端把“是否可用 AI”的检查抽成独立 hook/服务。设置页以路由状态携带返回目标，而不是把 API Key 或流程状态存到 URL、localStorage 或后端数据库。配置成功后继续首团创建；测试失败或用户取消则留在设置页，不创建会话。

## 错误处理

- 无可用 AI 配置：跳转设置并说明首团依赖 AI 守秘人。
- 示例内容缺失或不可用：显示“示例内容不可用”，提供前往模组页和新建游戏页的入口。
- 会话创建失败：不产生角色或会话残留，提示用户重试。
- 开场生成失败：继续复用会话页现有的错误分类与重试能力。

## 验收标准

1. 已配置 AI 的新用户可从首页一次操作进入一个新创建的自制示例单人会话。
2. 未配置 AI 的用户完成配置与测试后能回到并继续首团流程。
3. 手动新建会话、车卡和房间加入流程的行为不回归。
4. 首团编排具有后端单元/API 测试；前端关键交互有测试。
5. README、LICENSE 和内容声明准确描述当前功能与分发限制。
6. 后端测试、前端类型检查、构建与 lint 全部通过。

## 非目标

- 不实现离线故事或免费演示模型。
- 不把现有未确认授权的规则书或模组作为公开分发的示例内容。
- 不接入账户体系、公网鉴权或多进程实时架构。
- 不全面拆分 `chat_service.py` 或完整重写 `CharacterPage`。
