# 地图大改：调查板大地图 + 会生长的氛围现场图

**背景**：从玩家视角实测（鬼屋会话 890a2330）发现两张地图均沦为装饰：
场景地图是「好看的空壳」，与叙事两张皮；大地图根本不是地图，是一排 pill 按钮。

**已确认的方向**（2026-07-04 与用户对齐）：
- **大地图** → 调查板 / 侦探桌风格（节点 + 图钉 + 线，贴 CoC 气质与哥特主题）
- **场景图** → 氛围现场图（示意空间 + 调查交互 + 跟随叙事生长），**不做**战术棋盘

---

## 一、现状诊断（实测证据）

### 场景地图的问题（按伤害排序）

1. **与叙事两张皮（根源问题）**：叙事发生在「病房 + 走廊 + 半掩的门 + 灰影」，
   地图却是开局一次性生成的 11×7 空盒子（仅 2 床 1 桌 2 椅，零内部隔墙）。
   KP 嘴里的空间和图上的空间对不上，玩家没有理由点开地图。
2. **图标不可读**：床/窗/桌/柜全是深色小圆点（lucide 三选一：Box/Eye/Armchair），
   不悬停不知道是什么；「窗户」用眼睛图标。
3. **NPC 撞脸出戏**：维托里奥与加布里埃尔共用同一巫师帽精灵（byKind 默认素材），
   1920 疗养院里站着俩巫师；队友标记同格互相叠住。
4. **纯观赏零交互**：不能点物体调查、不能点地格移动。
5. 58° 透视纵向压缩明显，420px 视口一半是空地。烛光迷雾氛围是优点，保留。

### 大地图的问题

1. **零空间感**：pill 按钮列表，方位/远近/路线不存在，CoC 的「城市侦查感」缺失。
2. **粒度混乱**：「委托与准备」是章节不是地点；「老房子 / 老房子-地下室 / 老宅所在街区」
   三个按钮实为一处地方的三个尺度。
3. **无状态信息**：线索进度（clue_ledger 已有！）、队友位置（party_locations 已有！）
   都没上图——已有数据资产未变现。

### 一句话诊断

> D&D 战棋思维的瓦片图套在 CoC 调查模组上，且与叙事零同步。
> CoC 需要的是「侦探桌上的城市地图 + 图钉 + 红线」（战略层）
> 和「跟着剧情长出细节的现场图」（战术层）。

## 二、方案 A：大地图 → 调查板（InvestigationBoard）

### 视觉

深色软木板/皮革桌面质感（CSS 渐变即可，沿用哥特暗色 token），地点是**钉在板上的卡片**：

- **节点 = 地点卡**：图钉圆点 + 地点名 + 状态徽记
  - 当前所在：金色图钉 + 高亮描边（`--color-accent`）
  - 已访问：常规卡片
  - 听说过未去：虚线边 + 泛灰（探索欲）
  - 队友在此：卡片下挂成员小徽章（复用 SeatIcon 语义）
- **边 = 已知路径**：暗色「麻绳」虚线（非红线；红线留给 P3 线索关联）
- 玩家头像作图钉钉在当前地点卡上

### 技术

- `@xyflow/react` + `dagre` **已是项目依赖**（ModuleGraph.tsx 在用），零新轮子。
- 自动布局用 dagre，节点可拖（会话内不持久化，v1 不做位置存储）。
- 新组件 `apps/web/src/components/game/InvestigationBoard.tsx`，
  替换 GameSessionPage 的 pill 列表面板；点节点 → 复用现有 confirmTravel → travel 流。

### 数据（后端小改）

`list_known_locations` 增补：
- `connections: [scene_id]`（过滤到已知集合内——未知邻居不漏出去，防剧透）
- `party: [成员名]`（据 world_state.party_locations 归并）
- 场景新增可选字段 `kind: "location" | "chapter"`（默认 location）：
  `chapter` 不进大地图（解决「委托与准备」混入）。模组解析 prompt 顺带提示标注；
  存量模组无此字段 → 全部照旧显示（向后兼容）。
- 楼层型场景（老房子×地下室）**不归并**：在图上以边相连本身就是层级表达，
  比列表更清楚（v1 不引入 parent 关系，避免过度设计）。

## 三、方案 B：场景图 → 会生长的氛围现场图

保持现有 2.5D 烛光渲染底子，四步改造：

### P1' 可读性（纯前端）

1. 物体图标：按**名称关键词** → `react-icons/gi` 矢量图标映射（床/桌/椅/窗/柜/毯/门…
   各有形、分色），kind 兜底（furniture/item/feature 三档默认图标）。
2. **常显微标签**：物体与 NPC token 下方常显 9px 羊皮纸色小标签（带描边阴影），
   不再只靠悬停；缩放 <0.55 时隐藏防糊。
3. NPC 撞脸：同一素材被 ≥2 个人物 token 引用时，全部回退为「首字 + 唯一色环」
   （名字 hash → 色相），杜绝俩 NPC 一张脸。
4. 同格 token 扇形错开（±10px 水平位移按序分配）。

### P2 交互（前端 + 输入框预填）

MapView 增 `onIntent?: (text: string) => void` 回调，游戏页把它接到输入框**预填**
（不自动发送，玩家保有最终否决权——与 travel 二次确认同一哲学）：

- 点物体 → `我调查【床】`
- 点 NPC → `我走近维托里奥·马卡里奥`
- 点出口 → `我走向「通往走廊的门」`
- 点空地板 → `我走到房间东侧`（按格坐标相对地图中心算方位）

模组预览处不传回调 → 行为不变。

### P2' 叙事锚定 [MAP_MARK]（核心增量，后端为主）

KP 新增内联指令（完全仿 [MOVE] 的机制：就地插入、自动隐去、不打断行文、有图才广告）：

```
[MAP_MARK: name=半掩的门, near=走廊尽头/坐标/已有锚点, kind=feature|item|furniture]
```

- 解析落 `world_state.map_marks[scene_id] += [{name,x,y,kind}]`
  （near 复用 map_service._resolve_anchor + _nearest_floor 解析成坐标）。
- `current_scene_map()` 把 marks 合并进该场景 objects 下发 → 前端零改动自动渲染。
- 幂等：同名 mark 不重复添加（更新坐标）。
- KP 提示词（MAP_MARK_INSTRUCTION）与 MOVE_INSTRUCTION 同段注入，
  措辞强调「只标注叙事中**确实出现**的显著空间元素，别把一句气氛描写都钉上图」。
- **评估**：改 KP 提示词须跑 `python -m evals.run` 对比（tomb fixtures 带图，会注入本指令）。

这一步做完，地图才「活」：KP 说「走廊尽头有扇半掩的门」，图上就长出那扇门。

### P3

- 调查板**红线**：clue_ledger 线索 ↔ 地点关联可视化（红线连卡片）。✅ 已落地（P3a）
- 俯视 90° / 2.5D 视角切换。✅ 已落地（P3b）
- 生成质量校验：大建筑强制房间分割（隔断墙判定 + 重试一次）；
  出口语义（跨城出口不画成室内门，prompt 级）。✅ 已落地（P3c）
- 「已搜过」痕迹：调查过的物体变灰/打勾——**仍暂缓**（缺可靠的确定性来源）。
- AI 生图城市底图（远期锦上添花，不依赖）。仍搁置。

## 四、实施顺序与验收

| 期 | 内容 | 验收 |
|---|---|---|
| P1 | 调查板大地图 | 浏览器：节点图渲染、状态徽记正确、点卡前往可用；pytest 全过 |
| P1' | 场景图可读性 | 浏览器：图标分形分色、标签常显、NPC 不撞脸、不叠格 |
| P2 | 场景图点击交互 | 浏览器：四类点击各自预填正确文本；模组预览不受影响 |
| P2' | [MAP_MARK] | 单测（解析/落库/合并/幂等）+ eval 对比不回退 + pytest 全过 |

**不做**：不换渲染引擎（问题在信息设计不在渲染技术）；不做战棋精确规则；
大地图不上 AI 生图。

**硬约束**（开发守则）：一切图标用 `react-icons/gi` / `lucide-react` 矢量，禁 emoji；
图标由前端按语义渲染，后端不把图标塞进内容字符串。
