# 战斗方格位置模型 设计方案（CoC 7e 战斗轮）

> 承 [2026-07-11-战斗轮重构-design.md](2026-07-11-战斗轮重构-design.md) 系列（P1 可暂停状态机 + 反应 / P2 动作集 / P3 HUD 战场）。
> 本方案给现有**无空间 HUD 战场**叠加一层「真方格」空间模型（博得之门 3 式战棋），
> 只加规则与状态字段，**不改既有攻防/反应/两段式投骰/伤害结算的语义**。
>
> 目标产物：本设计文档。**不改任何代码**。

---

## 0. 一句话概括

在 `world_state.combat` 里加一块 `grid`（网格尺寸 + 障碍/掩体格）与每个参战方一个 `pos: {x,y}` 坐标；
新增一个**纯几何模块** `rules/coc/positioning.py`，把「距离 / 射程 / 抵近 / 夹击 / 掩体 / 视线」统一
**折算成既有的 `bonus`/`penalty` 奖惩骰**，交给现有 `resolve_attack` / `resolve_skill_check` 消费；
`combat_service` 只多做「布阵 + 移动结算 + 把几何折算喂给攻击」三件事。前端在 `CombatStage` 顶部
加一块纯 CSS-grid 棋盘（棋子 + 移动高亮 + 射程/威胁高亮），全走 `--color-*` 变量，不引第三方战棋库。

## 1. 约束与方向（务必先读）

产品负责人已拍板做**战棋式真方格**。但本项目此前删过「探索用瓦片小地图」，原因是
**「让几何去还原 KP 的自然语言叙事 → 几何欠定、地图与旁白脱节」**（见 MEMORY `project-map-feature`）。
本方案方向**相反**——几何被玩家操作出来、叙事跟随几何，据此定死四条边界：

| # | 边界 | 落地做法 |
|---|------|---------|
| C1 | 方格**只在战斗态**存在 | 网格随 `world_state.combat` 生灭；`_end_combat` 清 `combat` 时网格一并消失。非战斗态无任何坐标。 |
| C2 | 开战时由**引擎默认阵型 + KP 微调**布置初始站位，不追求还原真实房间尺寸 | `positioning.default_deployment()` 给一套「我方一侧 / 敌方一侧」的确定性站位；KP 可选覆盖，但**不解析旁白去测量房间**。 |
| C3 | 位置此后只由**离散移动动作**改变 | 唯一改坐标的入口是 `resolve_move()`（移动力 = 每轮 N 格）。没有连续拖拽、没有物理引擎。 |
| C4 | KP 叙事**跟随**棋盘状态，不反向驱动几何 | 棋盘是**唯一事实源**（single source of truth）；子代理叙述读棋盘状态生成文字，**绝不**从叙述回写坐标。 |

> 一句话记住反陷阱：**几何 → 叙事**（本方案），不是**叙事 → 几何**（被删的小地图）。
> 棋盘是玩家用离散动作「摆」出来的确定性状态，旁白是它的读者而非作者。

## 2. 网格数据结构（存 `world_state.combat`）

### 2.1 网格本体 `combat.grid`

在现有 `state`（`start_combat` 造的 dict：`active/round/turn_index/initiative/log/...`）里**平级加一个 `grid`**：

```jsonc
{
  "active": true, "round": 1, "turn_index": 0,
  "initiative": [ /* 见 2.2 */ ],
  "log": [], "started_seq": 128,

  "grid": {
    "cols": 12,                 // 列数（x: 0..cols-1）
    "rows": 8,                  // 行数（y: 0..rows-1）
    "cell_m": 1.5,              // 1 格 = 1.5 米（见 §3）
    "blocked": ["6,3", "6,4"],  // 完全阻挡格：不可进入、且阻断视线/射线（墙、巨物）
    "cover": { "5,2": "half", "5,3": "full" }  // 掩体格：half=半掩体, full=全掩体（见 §7）
  }
}
```

设计取舍：
- **坐标键用字符串 `"x,y"`**（不是嵌套数组），因为它要进 SQLite 的 JSON 列、要能被 set 去重、要好比较；
  几何计算时在 `positioning.py` 里 `"x,y" ↔ (x,y)` 互转（`_key`/`_xy` 两个私有工具）。
- `blocked` 与 `cover` 分开：`blocked` 是硬阻挡（连视线都断），`cover` 是可站/可穿但给遮挡加成。
- 网格**稀疏存障碍**：只存非空格，绝大多数空格不落库，省体积。

### 2.2 单位坐标——加在参战方 dict 哪里

坐标就加在**每个参战方 dict** 上（即 `initiative[]` 的元素），与既有 `hp/status/conditions/aim` 平级。
在 `_char_participant` / `_npc_participant` 的返回 dict 末尾各加两个字段：

```jsonc
{
  "id": "char_1", "name": "调查员甲", "side": "player",
  "hp": 11, "max_hp": 11, "status": "ok",
  "weapon": ".38(9mm)左轮", "has_firearm": true,
  "conditions": [], "aim": false,

  "pos": { "x": 2, "y": 3 },   // 新增：格坐标（开战由 default_deployment 落）
  "mov": 8                     // 新增：移动力基准（格数 = MOV，缺省 8，见 §4）
}
```

> 为什么放参战方 dict 而不是 grid：坐标是**单位的属性**，跟着单位排先攻、被 `advance_turn` 重排、
> 出局后仍留在原格（尸体/掩体语义）。放 grid 里再按 id 索引反而要两处同步。

### 2.3 前端投影 `_combat_meta`

`_combat_meta` 的 `order[]` 投影每项加 `pos`（`mov` 前端画移动高亮时要，也带上）；
`_combat_meta` 顶层加 `grid`（前端画棋盘要）。断线重连走 `GET /combat` 也就自动带上，无需额外通道。

```python
# _combat_meta 内，order 每项追加：
"pos": p.get("pos"), "mov": int(p.get("mov") or 8),
# _combat_meta 顶层追加：
"grid": state.get("grid"),
```

## 3. 网格尺度与射程换算

- **1 格 = 1.5 米**（`cell_m = 1.5`）。选 1.5m 与 BG3 / D&D 的 5ft 格同口径，也让「移动力 = MOV 格」
  落在合理的每轮位移（MOV 8 → 12m/轮，约一次疾走）。
- **默认棋盘 12×8 格**（18m × 12m），近身遭遇的抽象战术区，不是真实房间；KP 可在开战参数里改
  `cols/rows`（见 §5）。
- **武器射程（米）→ 格**：`range_cells = ceil(range_m / cell_m)`。`positioning.range_in_cells(weapon)` 解析
  武器表 `range` 字段：纯数字取米；`"接触"` → 1 格（近战，须相邻）；`"STR/5m"`/`"STRft"` 等投掷式按
  投掷者 STR 现算（`STR/5` 米再折格），MVP 可先回落一个保守常数（如 5 格）后续再精细化。

射程折格样例（`cell_m=1.5`）：

| 武器 | `range` | 米 | 格（`ceil(m/1.5)`） |
|------|---------|----|--------------------|
| 徒手格斗 | 接触 | — | 1（相邻） |
| 长鞭 | 3 | 3 | 2 |
| .38 左轮 | 15 | 15 | 10 |
| .30 杠杆步枪 | 50 | 50 | 34 |
| 加兰德 M1 | 110 | 110 | 74 |

> **刻意承认的现实**：12×8 的棋盘对角线才 ~14 格，手枪(10格)以上的火器在盘面上**几乎恒在射程内**。
> 这是**有意**的——盘面是「近身战术区」，真正产生几何张力的是**相邻(近战)、抵近(奖励骰)、掩体、夹击**，
> 而非 74 格的步枪射程上限。射程判定主要用来卡「投掷/短射程武器够不够得着」和「超程惩罚」。

## 4. 移动动作

### 4.1 移动力来源

`mov`（格/轮）= 角色 **MOV**（`system_data.MOV` 或角色卡派生值），缺省 **8**。
NPC dict 无 MOV 时同样回落 8（可选：`8 + (DEX≥80? +1 : DEX≤40? -1 : 0)` 的轻修正，MVP 不做）。

### 4.2 每轮走几格 —— 常规移动 vs 冲刺

沿用 CoC「可移动 MOV 并仍能行动」，但拆成两档给战术抉择（防「满移动 + 攻击」过强）：

| 档位 | 距离上限 | 能否再行动 |
|------|---------|-----------|
| **常规移动**（move） | `⌈mov/2⌉` 格（MOV 8 → 4 格） | **可**，移动后仍能攻击/技能等一个主动作 |
| **冲刺**（dash） | `mov` 格（MOV 8 → 8 格） | **否**，冲刺即用掉本回合动作 |

移动**不直接推进先攻**（`advance_turn` 只由主动作触发）；一个回合内：
`常规移动(可选) → 一个主动作` 或 `冲刺(独占) → 结束`。参战方 dict 加一个每回合预算
`move_left`（回合开始 = `mov`，`advance_turn` 走满一圈重置本轮标记时一并重置），移动扣格数、
主动作结算时若 `move_left < mov`（已走过）则本次移动视为「常规」档校验；实现细节见 §11。

### 4.3 对角、路径与占用

- **对角允许**，MVP 用 **Chebyshev 距离**（八方向每步算 1 格，斜走不额外收费）——最省心，先上；
  后续想更拟真可切「1-2-1」（斜走隔一步 +1）留作 Phase 后期开关。
- 目标格必须：在盘内、非 `blocked`、**未被其它存活单位占用**（一格一人）。
- 路径可达用 BFS（`positioning.reachable_cells`）：从 `pos` 出发、绕开 `blocked` 与占用格、步数 ≤ 预算。
  前端高亮可达格也用它（或前端自算，服务端仅做最终落点校验，见 §11 取舍）。

### 4.4 被近战牵制时的脱离

「脱离」= 从一个**与存活敌人相邻（距离=1）的格**移动到不再与之相邻的格。CoC RAW 无借机攻击，
本方案给一个轻量代价（对齐既有 grapple 语义，不引入全新机制）：

| 情形 | 规则 |
|------|------|
| 被 `grappled`（已被擒抱） | **不能移动**（先做 `maneuver` 挣脱或其它，与现有「擒抱隐藏逃跑」一致） |
| 与敌相邻但未被擒，选择脱离 | 只能走**常规移动**档，且**本回合不能再攻击**（脱离即用掉动作，等价「小步撤离」）；不触发额外攻击 |
| 与敌相邻，选择冲刺脱离 | 允许，冲刺本就独占动作；正常按冲刺距离撤 |

> 取舍：不做「借机攻击」是因为它需要在别人回合插入一次攻击结算，跟现有「反应仅由 NPC 攻击真人触发」
> 的暂停模型不合拍；用「脱离即弃攻击」达到「贴身缠斗不好走开」的手感，且零新机制。

## 5. 开战初始布阵

`start_combat` 造完 `state` 后、`roll_initiative` 前后（顺序无所谓，坐标与先攻正交），调用
`positioning.default_deployment(participants, grid)` 原地给每个参战方落 `pos`：

**默认阵型**（确定性、可测）：
- 我方（`player`/`ally`）贴**左**两列（x = 1, 2），敌方（`enemy`）贴**右**两列（x = cols-2, cols-3）；
  中间留空当交火区。
- 同侧单位沿 y 轴**均匀铺开并居中**：n 人时占据 `rows` 中间的 n 行（或双列折行），间隔 ≥1 行防扎堆。
- 纯确定性排布（按 `initiative` 顺序或 id 稳定序落位），便于单测钉死。

```python
def default_deployment(participants: list[dict], grid: dict) -> None:
    """原地给每个参战方落 pos：我方左侧两列、敌方右侧两列，各自沿 y 居中均匀铺开。
    纯确定性（按传入顺序），不掷骰、不读叙事。"""
```

**KP / 引擎微调（可选，不还原房间）**：
- `start()` / `start_combat()` 增一个可选入参 `deployment: dict | None`（`{char_id: {"x":..,"y":..}}`），
  非空则覆盖默认落点（越界/占用/落在 blocked 的非法项忽略、回落默认）。KP 想「敌人从门口涌入」只是
  把敌方 pos 设到某列，**不是**去解析「门在房间哪面墙」——仍是几何被摆出来，不是叙事被测量。
- 也可留一个开战首回合的「repose」窗口（KP 拖动几步）后续再加，MVP 用默认阵型即可。

## 6. 射程判定

`positioning.range_check(distance_cells, range_cells, ranged) -> (bonus, penalty, reachable)`：

| 距离 vs 射程 | 结果 | 折算 |
|-------------|------|------|
| 近战武器（`range="接触"`），距离 ≤ 1 | 可打 | 正常（近战对抗，见 §10） |
| 近战武器，距离 > 1 | **够不着** | `reachable=False` → 前端禁该攻击，提示「需先移动接近」 |
| 火器，距离 ≤ `range_cells` | 在射程内 | 正常检定 |
| 火器，`range_cells` < 距离 ≤ `2×range_cells` | 超基础射程 | **-1 惩罚骰**（可叠加 §7 掩体） |
| 火器，距离 > `2×range_cells` | 超出可及 | `reachable=False` → 不可射击 |

`reachable=False` 时服务端拒绝该攻击（`raise ValueError("目标超出射程/够不着")`），前端在下拉/棋子上灰掉。
`penalty` 直接进 `resolve_attack` 的 `penalty`（见 §11）。

## 7. 抵近射击（产品重点）

**核心张力**：贴近了打**更准**（奖励骰），但站进了敌人拳脚范围会被**夹击**、事后闪避**更险**（惩罚骰）。
把「奖励」绑在**距离**（几何）、把「代价」绑在**相邻敌数**（几何），两者天然由棋盘位置产出：

### 7.1 抵近奖励（命中）

`positioning.point_blank_bonus(distance_cells, ranged) -> int`：

| 用火器时，与目标距离 | 命中奖励 |
|--------------------|---------|
| ≤ 2 格（≈3m，抵近） | **+1 奖励骰** |
| 3 格及以上 | 0 |

> 近战武器不吃此项（近战本就相邻）。奖励骰经 `resolve_attack(bonus=...)` 透传（与现有「瞄准 +1」同管道，
> 可叠加：抵近 + 瞄准 = +2 奖励骰）。

### 7.2 抵近代价（夹击惩罚，见 §8）

奖励只看距离；**代价来自「你是否与敌相邻」**。要打出 ≤2 格的抵近奖励，常常得走到与敌 **相邻（1 格）**，
于是陷入近战接触——此后被攻击选闪避时吃 §8 的夹击惩罚。**站在正好 2 格**（有奖励、不相邻）是「老练的
抵近走位」，站在 1 格是「贴脸对拼」（奖励 + 夹击风险）。这条 2 格 vs 1 格的分界就是抵近的战术深度。

量化速查：

| 站位 | 抵近命中奖励 | 被近战攻击时闪避 |
|------|:-----------:|:---------------:|
| 2 格（不相邻） | +1 奖励骰 | 无夹击惩罚 |
| 1 格 / 1 名敌相邻 | +1 奖励骰 | 无额外（首个相邻敌不罚，见 §8） |
| 1 格 / 2 名敌相邻 | +1 奖励骰 | **-1 惩罚骰** |
| 1 格 / ≥3 名敌相邻 | +1 奖励骰 | **-2 惩罚骰**（封顶） |

## 8. 夹击 / 腹背受敌

`positioning.flank_penalty(defender, participants) -> int`：数**与防御者相邻（距离=1）的存活敌人**，

```
adj = 相邻存活敌人数
夹击惩罚骰 = max(0, adj - 1)，封顶 2
```

- 1 名相邻敌：0（单挑不罚）；2 名：-1；≥3 名：-2。
- 该 `penalty` 加到防御者的**闪避/招架检定**上：
  - NPC 防御（`_apply_one_action` 攻击分支）→ 折进传给 `resolve_attack` 的 `penalty`；
  - 真人反应（`resolve_reaction` 的 dodge/fight_back）→ 同样折进 `resolve_attack(penalty=...)`。
- 与 §7 抵近共同构成「贴脸打枪但会被围殴」的闭环。

## 9. 连发扫射多目标（位置层供料）

连发的「换目标 +1 惩罚骰」正在**另线**实现，本方案**只负责供料**——回答「哪些单位能被一次连射扫到」：

`positioning.sweep_targets(shooter_pos, primary_target, participants, grid) -> list[dict]`：
- 返回可被同一次扫射覆盖的目标集：以主目标为锚，**彼此相邻（成排/相邻格）**且与射手大致同向、有视线的敌人。
- MVP 判据：主目标 + 与主目标**距离 ≤ 1 格**的其它存活敌人（成排扫）。后续可精细成「射手→主目标射线
  两侧 ±1 格的锥形」。
- 位置层只产出**候选目标列表**（几何事实）；每多打一个目标 +1 惩罚骰、弹药消耗等由连发线自己按这份
  名单叠加。契约：`sweep_targets` 纯函数、不改状态、不碰惩罚骰数值。

---

## 10. 近战接触 = 相邻格

- 近战攻击（`range="接触"` 或非火器武器）**要求攻防相邻**：`positioning.cell_distance(a, b) <= 1`（Chebyshev，
  含对角）。不相邻 → 该近战攻击 `reachable=False`，前端灰掉、服务端拒绝。
- 现有近战对抗 / 反击 / 擒抱缴械（`resolve_attack` 的 fight_back 分支、`resolve_maneuver`）语义**不变**，
  只是多一道「必须相邻」前置校验。
- `maneuver`（擒抱/缴械）同样要求相邻。

## 11. 与现有引擎 / service 的接口改动（尽量小增量）

原则：**几何计算全下沉到新纯模块 `positioning.py`，一律折算成 `bonus`/`penalty` 奖惩骰**，喂给

```python
def cover_penalty(a: dict, b: dict, grid: dict) -> int:               # 半掩体 -1；全掩体经 LOS 断→上层判不可命中
def default_deployment(participants: list[dict], grid: dict) -> None: # 原地落 pos（确定性）
def reachable_cells(start: dict, budget: int, grid: dict, occupied: set[str]) -> set[str]:  # BFS 可达格
```

**攻击几何折算的组合方式**（在 service 里组合，引擎仍只认最终 bonus/penalty）：

```python
# combat_service 内，攻击结算前统一算一次：
dist = positioning.cell_distance(actor, target)
range_cells = positioning.range_in_cells(weapon, grid["cell_m"])
r_bonus, r_penalty, reachable = positioning.range_check(dist, range_cells, ranged)
if not reachable:
    raise ValueError("目标超出射程/够不着，请先移动接近")
if ranged and not positioning.has_line_of_sight(actor, target, grid):
    raise ValueError("没有射击视线（被遮挡）")
hit_bonus   = (1 if actor.get("aim") else 0) + positioning.point_blank_bonus(dist, ranged) + r_bonus
hit_penalty = r_penalty + positioning.cover_penalty(actor, target, grid)   # 攻方命中侧
res = engine.resolve_attack(..., bonus=hit_bonus, penalty=hit_penalty)      # ← 见 11.2
```

防御侧（守方闪避/招架）的夹击惩罚，在 `resolve_attack` 内部对**守方检定**施加，或由 service 预算后传入——
见 11.2 的 `defense_penalty` 入参。

### 11.2 `rules/coc/combat.py::resolve_attack` —— 最小签名增补

现签名已有 `bonus: int = 0`（喂攻方命中），但**缺 `penalty`**。增补两个可选入参，默认 0 时行为与旧版逐字节一致：

```python
def resolve_attack(
    attacker_data, attacker_db, weapon, *,
    defender_data=None, defense=None, ranged=False, difficulty="normal",
    attacker_disarmed=False,
    bonus: int = 0,
    penalty: int = 0,            # 新增：攻方命中检定的净惩罚骰（超程/掩体）→ 透传 resolve_skill_check
    defense_penalty: int = 0,    # 新增：守方闪避/招架检定的惩罚骰（夹击）→ 施于守方那次 resolve_skill_check
) -> dict:
```

改动点（都在既有分支里补参，不新增分支）：
- 攻方所有 `resolve_skill_check(attacker_data, atk_skill, ..., bonus=bonus)` 调用补 `penalty=penalty`
  （含远程「扑掩体转困难」重掷那次）。
- 守方 `resolve_skill_check(defender_data, "闪避"/dfn_skill, "normal")` 补 `penalty=defense_penalty`。
- **纯增量**：两参默认 0 → 现有全部单测断言不变（旧调用不传即旧行为）。

### 11.3 `combat_service.py` 改动清单

| 函数 | 改动 |
|------|------|
| `_char_participant` / `_npc_participant` | 返回 dict 末尾加 `"pos": None`（开战时 default_deployment 落）、`"mov": MOV or 8` |
| `start_combat` | 造 `state` 时加 `grid`（默认 12×8 或按入参）；调 `positioning.default_deployment`；可选 `deployment` 覆盖 |

| `_apply_one_action`（attack 分支） | 攻击前算 §11.1 折算，`reachable`/LOS 不满足则记「够不着/无视线」摘要不结算；否则把 `bonus/penalty/defense_penalty` 传入 `resolve_attack` |

### 11.4 端点 / schema

- `CombatActionRequest` 加 `dest: dict | None`（`{"x":int,"y":int}`），并允许 `type="move"`（与连发线
  已加的 `shots` 字段并存，互不相干）：
  ```python
  class CombatActionRequest(BaseModel):
      type: str = "attack"
      target_id: str | None = None
      weapon: str | None = None
      defense: str | None = None
      kind: str | None = None
      shots: list[str] | None = None    # 已存在：连发多目标（连发线）
      dest: dict | None = None          # 新增：type="move"/"dash" 的落点 {x,y}
  ```
  > 注：`shots`（连发每发目标序列）是 §9「连发扫射」另线已落的字段——位置层的 `sweep_targets` 正是给它
  > 供「哪些相邻敌可入 `shots` 名单」的几何候选，两者契约互补。
- `POST /combat/action` 里 `type in ("move","dash")` 路由到 `combat_service.resolve_move`（其余照旧走
  `resolve_player_action`）。移动**不消耗先攻回合**，广播新 `combat_state` 后玩家可继续在同回合选主动作。

## 12. 前端 HUD 方格渲染草图

在 `CombatStage` 现有「先攻轨 + 两栏卡片」**上方**插一块棋盘 `<CombatGrid>`（两栏卡片保留作详情/血条，
棋盘负责空间）。纯 **CSS Grid + 绝对定位棋子**，无第三方战棋库，全部颜色走 `--color-*`（gothic/parchment 自适应）。

### 12.1 结构

```tsx
// combat.grid: { cols, rows, cell_m, blocked: string[], cover: Record<string,'half'|'full'> }
// order[]: 每项含 pos:{x,y}, mov, side, hp/max_hp, status...
function CombatGrid({ grid, order, turn, myCharId, mode, reachable, onCellClick, onPieceClick }) {
  const CELL = 34 // px，随容器/断点缩放；mobile 用更小
  return (
    <div className="relative mx-auto" style={{
      width: grid.cols * CELL, height: grid.rows * CELL,
      display: 'grid',
      gridTemplateColumns: `repeat(${grid.cols}, ${CELL}px)`,
      gridTemplateRows: `repeat(${grid.rows}, ${CELL}px)`,
      // 棋盘网线：细琥珀边（复用 --color-border）
      backgroundImage:
        `linear-gradient(var(--color-border) 1px, transparent 1px),
         linear-gradient(90deg, var(--color-border) 1px, transparent 1px)`,
      backgroundSize: `${CELL}px ${CELL}px`,
      border: '1px solid var(--color-border-strong)',
      background: 'var(--color-bg-secondary)',
    }}>
      {/* 1) 障碍/掩体格：blocked 实心暗块；cover half/full 斜纹或砖墙图标(GiBrickWall) */}
      {/* 2) 高亮层：move 模式下 reachable 格铺 accent 半透明；attack 模式下敌方射程/威胁铺 danger 半透明 */}
      {cells.map(({x,y,kind}) => (
        <button key={`${x},${y}`} onClick={() => onCellClick(x,y)}
          style={{ gridColumn: x+1, gridRow: y+1,
            background: kind==='reach' ? 'color-mix(in srgb, var(--color-accent) 22%, transparent)'
                      : kind==='threat' ? 'color-mix(in srgb, var(--color-danger) 20%, transparent)'
                      : 'transparent' }} />
      ))}
      {/* 3) 棋子层：绝对定位到格中心；阵营色（我方 --color-accent / 敌方 --color-danger）、
             当前行动者外发光（复用卡片 active 的 box-shadow）、出局灰化、小血条 */}
      {order.map(c => c.pos && (
        <button key={c.id} onClick={() => onPieceClick(c)}
          className={c.id===turn ? 'ring-active' : ''}
          style={{ position:'absolute',
            left: c.pos.x*CELL, top: c.pos.y*CELL, width: CELL, height: CELL,
            /* 圆形棋子 + 阵营色描边 + 首字/SeatIcon */ }}>
          {/* Gi* 图标或名字首字；下方 2px 血条 width=hp/max_hp% */}
        </button>
      ))}
    </div>
  )
}
```

### 12.2 交互与高亮

- **模式 `mode`**：`'idle' | 'move' | 'attack'`（由动作栏当前选中动作决定）。
- **移动高亮**：进入 `move`/`dash`，把 `reachable`（服务端 `reachable_cells` 或前端同算的 BFS）铺
  `--color-accent` 半透明；点可达格 → `POST /combat/action {type:'move', dest:{x,y}}`。
- **射程/威胁高亮**：`attack` 模式选中我方棋子，铺当前武器射程内的格（`--color-accent` 淡）与
  会被敌方近战威胁的格（`--color-danger` 淡）；点敌方棋子 = 选目标（等价现有目标下拉）。
- **点棋子**：己方棋子→选为行动者视角；敌方棋子→设为攻击目标（与现有 `targetId` 状态打通，棋盘与
  下拉双向同步，保留下拉作无障碍回退）。
- **动画**：移动用 CSS `transition: left/top .18s`；沿用现有 `hp-hit`/`hp-float` 掉血动画；
  `prefers-reduced-motion` 时棋子瞬移、跳过。
- **主题**：全程 `--color-*` + `color-mix`，gothic（暗底琥珀线）与 parchment（羊皮纸底深红线）零改自适应。

### 12.3 布局取舍

- 棋盘默认展开在面板顶部；窄屏（mobile ~380px）CELL 降到 ~22px、或棋盘可折叠（复用现有 `collapsed`）。
- 两栏卡片**保留**（棋子太小放不下完整状态/条件徽标），棋盘给空间感、卡片给细节，二者共享同一 `order` 数据。

## 13. 分期实施建议

| 阶段 | 范围 | 交付 |
|------|------|------|
| **MVP（P-Grid-1）** | 站位 + 移动 + 射程可达 + 近战接触 | `positioning.py`（`cell_distance/is_adjacent/range_in_cells/range_check/default_deployment/reachable_cells`）；`grid` 落库 + `pos/mov` 字段；`resolve_move` + `type=move` 端点；`resolve_attack` 加 `penalty`（先只用于超程 -1）；近战不相邻/超程 → 拒绝；前端 `CombatGrid` 画棋子 + 移动高亮 + 点击移动/选目标 |
| **P-Grid-2** | 抵近奖励 + 夹击惩罚 | `point_blank_bonus` / `flank_penalty` 接入攻方 bonus 与守方 defense_penalty；前端威胁高亮 |
| **P-Grid-3** | 掩体 + 视线 | `cover`/`blocked` 格、`has_line_of_sight` / `cover_penalty`；全掩体不可命中、半掩体 -1；棋盘画掩体格 |
| **P-Grid-4** | 连发扫射供料 + NPC 走位打磨 + 冲刺档 | `sweep_targets` 供连发线；NPC 启发式「够不着先靠近」；`dash` 档与脱离牵制细则 |

> MVP 先做「站位 + 移动 + 射程 + 近战接触」正是任务建议的最小可用集：它已能让战斗**有空间**（走位、
> 拉近拉远、近战必须贴上去），且不碰任何奖惩骰数值平衡（超程 -1 是唯一新惩罚），风险最低。
> 抵近/夹击/掩体都是在这个骨架上**叠 bonus/penalty**，天然增量、可单测、可回滚。

**验证纪律**（每阶段）：
- 后端 `cd server && .venv/bin/pytest -q && .venv/bin/ruff check .` 全绿；`positioning.py` 纯函数补单测
  （距离/射程折格/布阵确定性/BFS 可达/夹击计数/LOS），几何用钉死坐标断言，不掷骰。
- 前端 `cd apps/web && npx tsc --noEmit && npx vite build`。
- `resolve_attack` 加参后**旧战斗单测断言不变**（默认 0 = 旧行为）。

## 14. 衍生到非战斗（简短设想）

位置模型可**极克制地**外溢到非战斗，但必须守住 §1 的反陷阱——**仍是玩家用离散动作摆出的确定性状态，
不是用几何还原 KP 旁白**：

- **抽象「站位 / 区域」而非瓦片图**：社交/潜行场景用少量**命名区域**（门口/吧台/阴影里）或粗格（如 4×4），
  玩家显式「移动到 X 区」，供潜行(躲进阴影=区域给 Stealth 奖励骰)、伏击(先手)、社交距离等判定。
- **复用同一 `positioning` 折算**：区域相邻/视线 → 同样折成奖惩骰喂给通用 `resolve_skill_check`，零新机制。
- **同样只在「有位置意义的场景」临时开启**，结束即清——与战斗态同生灭的做法一致，绝不做常驻探索大地图
  （那是已删的、几何欠定的老路）。

这一步**不在本方案实施范围内**，仅记录方向：先把战斗方格做扎实，非战斗复用等有真实需求再谈。

## 附：关键设计决策速查

- **网格尺度**：1 格 = 1.5 m；默认棋盘 12×8；射程格 = `ceil(米/1.5)`；长枪在小盘上近似恒在射程内（有意）。
- **坐标存哪**：每个参战方 dict 加 `pos:{x,y}` + `mov`（与 `hp/conditions` 平级，进 `initiative[]`）；
  网格本体 `combat.grid = {cols,rows,cell_m,blocked[],cover{}}`；`_combat_meta` 投影带上，重连自动对齐。
- **抵近射击量化**：火器距离 ≤2 格 → 命中 **+1 奖励骰**；代价来自**相邻敌数** → 闪避 `-(相邻敌数-1)` 惩罚骰
  （封顶 -2）。站 2 格=纯奖励，站 1 格=奖励但吃夹击。
- **MVP 范围**：站位 + 移动(`⌈mov/2⌉`常规/`mov`冲刺) + 射程可达 + 近战须相邻；奖惩上只引入超程 -1。
- **主要接口改动**：新增纯模块 `rules/coc/positioning.py`（几何→奖惩骰）；`resolve_attack` 加
  `penalty`/`defense_penalty`（默认 0，旧行为不变）；`combat_service` 加 `resolve_move` + 布阵 + 折算喂给
  攻击；`CombatActionRequest` 加 `dest` + `type=move`；前端 `CombatStage` 加纯 CSS-grid `CombatGrid`。
